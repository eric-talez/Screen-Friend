Roadmap v2 · Screen Friend / 화면 친구

Mac 화면 위에 사는
작은 AI Companion

Screen Friend는 사용자의 실제 Mac 화면 위에서 조용히 살아 움직이는 desktop companion 앱이다. 단순한 웹 데모나 AI 3D 생성기가 아니라, 항상 위에 떠 있지만 작업을 방해하지 않는 귀여운 캐릭터 경험을 먼저 완성하고, 이후 사진 기반 AI 커스텀 캐릭터로 확장한다.

✅ Slice 0–8.5 merged 🎯 Slice 9: AI character (plan first) Mac-first Electron overlay Click-through companion Future: photo → cute character

1. 최종 제품 그림

완성된 Screen Friend는 화면 하단에 작은 캐릭터가 항상 떠 있고, 내가 Chrome·VS Code·PDF·Notion을 쓰는 동안 아래 앱 클릭을 방해하지 않는다. 캐릭터는 혼자 걷고, 쉬고, 눕고, 졸고, 잠들고, 마우스가 가까이 오면 짧게 반응한다. 메뉴바에서 숨기기·보이기·종료·설정 변경을 할 수 있고, 나중에는 사진을 넣어 나만의 캐릭터를 만들 수 있다.

방향성: “화면 안에 앱이 하나 떠 있다”가 아니라 “내 화면 위에 작은 생명체가 산다”는 느낌. 최근 영상 레퍼런스처럼 화면 하단·창 경계·작업 공간과 자연스럽게 공존하는 companion이 목표다.

Core UX

방해하지 않는 존재감

항상 위에 보이지만 click-through가 기본이라 아래 앱을 그대로 사용할 수 있다. 사용자는 캐릭터를 “앱 창”보다 “화면 위 친구”로 느껴야 한다.

Behavior

조용한 생동감

idle, walk, lie, sleepy, sleep, stretch, react가 자연스럽게 이어지고, 마우스 근접 반응은 짧고 차분해야 한다.

Personalization

나만의 캐릭터

장기적으로 사람·동물·캐릭터 사진을 귀여운 companion asset으로 변환해 Screen Friend에 적용한다.

3. Full Roadmap v2

Pivot 0

Project Reframe: AI 3D Demo → Screen Friend

목표: 제품의 중심을 “AI 생성 웹 데모”가 아니라 “Mac 화면 위에 사는 companion”으로 재정의한다.

  • README, CLAUDE.md, docs 방향 정리.
  • 기존 3D/customization mock은 삭제하지 않고 future prototype으로 격리.
Slice 1

Character Behavior Sandbox

목표: 웹에서 캐릭터가 살아 움직이는 최소 행동 loop를 만든다.

  • idle, walk, blink, lie, sleepy, sleep.
  • state, timer, direction, simple movement.
Slice 2

Electron Desktop Shell

목표: Vite/React renderer를 Electron desktop app 안에서 실행한다.

  • main process, preload, renderer 연결.
  • dev/build script 정리.
Slice 3

Transparent Always-on-Top Overlay

목표: 실제 Mac 화면 위에 캐릭터만 떠 보이는 투명 overlay를 만든다.

  • transparent, frameless, always-on-top.
  • bottom-of-screen positioning.
  • visible on fullscreen/workspaces 동작 실기기 QA 필요.
Slice 4

Click-through + Interactive Debug Mode

목표: 평소에는 아래 앱 클릭을 방해하지 않고, 개발 모드에서는 overlay를 클릭 가능하게 만든다.

  • normal mode: click-through.
  • interactive mode: debug badge + clickable overlay.
Slice 5

Behavior Scheduler v1 + Mouse Reaction

목표: weighted random scheduler 위에 calm mouse-near reaction을 추가한다.

  • react는 interrupt-only. 스케줄러 weight에는 넣지 않음.
  • sleepy/sleep은 방해하지 않음.
  • cooldown + pointer freshness로 스팸 방지.
Slice 6

Menu Bar / Tray Controls

목표: 메뉴바에서 Show, Hide, Interactive Mode, Quit을 조작한다.

  • 🐾 placeholder tray icon (실제 아이콘은 Slice 8 에셋에서).
  • runtime click-through / interactive toggle.
  • preload bridge는 최소 IPC만 노출.
  • tray 메뉴 실제 클릭·배지 토글·click-through 비회귀는 실기기 수동 QA 1회 필요.
Slice 7

Persistence & User Preferences

목표: 앱을 껐다 켜도 설정이 유지되게 한다.

  • interactive 모드 + 창 위치 영속화.
  • scale/personality/intensity schema 예약 (조작 UI는 후속).
  • 설정 파일: ~/Library/Application Support/@ai-3d-demo/desktop/screen-friend-settings.json
Slice 8

Character Asset Registry

목표: 향후 sprite sheet·GLB asset이 behavior engine을 건드리지 않고 플러그인 가능한 registry를 만든다.

  • action → CSS class + label 매핑을 versioned registry(characterAssets.ts)로 추출.
  • 기본 캐릭터: ”default-css”. 실제 sprite sheet는 후속 슬라이스.
Slice 8.5

Animation Polish & Surface Awareness

목표: 영상 레퍼런스처럼 “화면 위에 사는 느낌”을 강화한다.

  • 이동 ease-out, ground contact shadow로 화면 하단 anchoring.
  • 걷기 중 leg/tail 애니메이션, lie/sleep flat-pose 수정, sleepy sway 추가.
  • 화면 가장자리 근처 near-edge class 적용.
  • macOS Accessibility permissions 없음, 실제 window detection 없음.
Slice 9

Optional AI Custom Character

다음 — 반드시 sub-slice 분리 후 시작

목표: 사용자가 사진을 업로드하면 Screen Friend용 귀여운 캐릭터 asset으로 변환한다.

  • 사람/동물/캐릭터 reference → 귀여운 mascot companion.
  • 저작권/초상권 이슈를 피하기 위해 원본 복제가 아니라 스타일화된 변환.
  • 초기에는 2D sprite 생성, 나중에 GLB/3D 옵션 검토.
Slice 10

Packaging, QA, Beta

목표: 개발 명령어 없이 실행 가능한 Mac 앱으로 만든다.

  • macOS packaging, signing, notarization 조사.
  • Mission Control, fullscreen, multi-monitor, CPU/GPU QA.
  • 초기 베타 사용자 피드백 루프.
Recommended path

4. 2D 먼저, 3D는 나중

영상 같은 3D 느낌은 매력적이지만, MVP 이후 바로 3D GLB로 가면 rigging, animation, 투명 overlay 성능, packaging QA가 급격히 어려워진다. 따라서 첫 커스텀 캐릭터는 2D sprite 기반을 추천한다.

  • 2D sprite: 빠름, 예측 가능, 성능 좋음, AI 생성과 연결 쉬움.
  • 3D GLB: 영상에 더 가까움, 하지만 rig/clip/lighting/performance 비용 큼.
  • 전략: Slice 8에서 asset pipeline 추상화 → 2D 먼저 → 3D는 고급 옵션.
Product rule

5. AI 생성은 핵심 MVP 이후

이 앱의 핵심은 “AI가 예쁜 캐릭터를 생성한다”가 아니라 “생성된 캐릭터가 실제 화면 위에서 귀엽게 산다”이다. 따라서 AI 기능은 asset pipeline과 persistence가 준비된 뒤 붙이는 것이 안전하다.

  • Slice 7 전: 저장할 곳이 없음.
  • Slice 8 전: 생성된 asset을 앱에 안정적으로 적용할 구조가 없음.
  • Slice 9에서야 photo-to-friend가 제품적으로 자연스러워짐.

6. Photo → Screen Friend 변환 파이프라인

장기적으로 사용자가 사람·반려동물·캐릭터 이미지를 넣으면, Screen Friend가 원본을 그대로 복제하지 않고 귀여운 companion 스타일로 재해석해 앱에 적용한다.

1. Upload 사람, 동물, 그림, 캐릭터 reference 이미지 업로드.
2. Safe stylize 원본 특징을 살리되 저작권/초상권 위험을 낮춘 mascot 스타일로 변환.
3. Generate poses idle, walk, sleep, react 등 상태별 pose 또는 sprite sheet 생성.
4. Import asset Slice 8 asset pipeline으로 검증·크롭·배경 제거·스케일 조정.
5. Live companion 사용자의 화면 위에서 새 캐릭터가 기존 behavior engine에 연결되어 움직임.

안전한 방향: “피카츄 그대로”가 아니라 “노란색 작은 전기 느낌의 귀여운 화면 친구”처럼 일반화된 특징을 사용. 실제 사람 사진은 본인 또는 동의가 있는 이미지 중심으로 제한하는 것이 좋다.

8. 구현 원칙

계속 지킬 것

  • One task = one slice.
  • 작고 reviewable한 diff.
  • backend, database, auth, billing은 MVP에서 제외.
  • renderer는 Node API 직접 사용 금지.
  • Electron IPC는 preload의 최소 API만 노출.
  • 실행하지 않은 검증은 실행했다고 말하지 않기.

조심할 것

  • AI custom character를 너무 빨리 붙여 scope 폭발시키지 않기.
  • 3D GLB로 바로 전환해 animation/rigging 지옥에 빠지지 않기.
  • macOS Mission Control, fullscreen, multi-monitor 동작은 반드시 실기기 QA.
  • 저작권 캐릭터/타인 얼굴을 그대로 복제하는 생성 기능은 피하기.

9. Slice 9 시작 전 필수 계획

Slices 0–8.5가 완료되었다. Slice 9(Optional AI Custom Character)는 scope가 크므로, 구현 전 반드시 sub-slice로 쪼개고 각 sub-slice를 별도 PR로 진행해야 한다.

⚠️ Slice 9 전체를 한 번에 구현하지 않는다. 아래는 분리 예시일 뿐이며, 실제 분리는 별도 설계 슬라이스에서 결정한다. 전체 sub-slice 분리 계획 및 safety 규칙: docs/custom-character-plan.md

Goal: Plan Slice 9 — split into sub-slices before any AI implementation.

Before writing code:
- Define what "AI custom character" means in concrete UX terms.
- Decide on 2D sprite vs GLB as first target format.
- Identify which AI provider/API will be used and document why.
- Split into at least: (a) upload + validation UI, (b) API integration,
  (c) asset import into registry, (d) character switch at runtime.
- Each sub-slice = one PR, one typecheck+build pass.

Do not add AI, upload, backend, database, or auth until sub-slices are planned.
Do not combine this with Slice 10 packaging work.
