Project Plan · Desktop Companion MVP

Screen Friend / 화면 친구

Mac 화면 아래에 귀여운 캐릭터가 항상 위에 떠 있으면서 걸어다니고, 눕고, 졸고, 눈을 깜빡이는 desktop screen companion 앱. AI 3D 생성은 핵심 MVP가 아니라, 나중에 “내 캐릭터 만들기” 기능으로 확장한다.

Electron-first MVP Transparent always-on-top overlay Cute behavior loop Mac-first, cross-platform later

1. 최종 목표 정의

한 줄 설명: Screen Friend는 사용자의 실제 Mac 화면 위에 사는 귀여운 캐릭터다. 앱 창처럼 보이지 않고, 화면 아래쪽에 자연스럽게 떠 있으며, 사용자가 일하는 동안 조용히 움직이고 쉬고 졸고 반응한다.

중요: 기존 “AI 3D Demo”는 웹페이지 안에 MacBook mockup을 만들고 그 안에 캐릭터를 보여주는 프로토타입이었다. 새 목표는 실제 desktop overlay이므로, 다음 단계부터는 Electron 기반 desktop companion 방향으로 pivot한다.

Core experience

사용자가 느껴야 하는 것

  • 캐릭터가 내 실제 화면에 사는 것처럼 느껴진다.
  • 항상 위에 보이지만 작업을 방해하지 않는다.
  • 화면 아래를 천천히 걷고, 멈추고, 눕고, 졸고, 눈을 깜빡인다.
  • 가끔 마우스 근처에서 반응하거나 사용자를 바라본다.
  • 설정은 메뉴바/tray에서 간단히 켜고 끈다.
Non-goals for MVP

초기 MVP에서 하지 않을 것

  • 이미지 업로드 → AI 3D 생성 → 실시간 변환.
  • Meshy/API/backend/task polling.
  • 완성도 높은 App Store 배포.
  • 복잡한 캐릭터 커스터마이징 UI.
  • 처음부터 완벽한 3D rig/animation.

2. 기술 선택

선택지판단이유주의점
Electron MVP 추천 기존 React/Vite 자산을 재사용하기 쉽고, BrowserWindow로 frameless/transparent/always-on-top/click-through류 desktop window 제어가 가능하다. 앱 용량이 커질 수 있다. macOS 권한/창 레벨/전체화면 workspace 동작은 실제 기기 테스트 필요.
Tauri 나중 후보 가볍고 Rust 기반이라 배포 앱으로 매력적이다. 웹 프론트엔드도 재사용 가능하다. macOS transparent window가 private API feature와 연결되어 App Store 수용성 문제가 있을 수 있다. 초기 Codex 구현 난이도도 Electron보다 높다.
웹앱만 사용 최종 목표 불가 브라우저 내부 데모는 가능하다. 실제 Mac 화면 위 always-on-front overlay, click-through, tray control을 구현할 수 없다.

근거: Electron 공식 문서는 frameless window, transparent window, always-on-top, click-through mouse ignore, tray menu를 지원한다. Tauri도 window customization과 always-on-top 설정을 지원하지만, macOS transparent window는 private API 제약을 명시한다. 자세한 링크는 문서 하단 Sources 참고.

4. 핵심 시스템 설계

Overlay Window Layer

  • transparent + frameless window
  • always-on-top
  • bottom-of-screen positioning
  • visible on all workspaces 옵션
  • click-through mode: 기본은 화면 조작 방해 최소화
  • character hit-area만 선택적으로 interactable

Character Renderer

  • MVP: 2D sprite 또는 simple CSS/SVG/Canvas character
  • 확장: Three/R3F + GLB character
  • 상태별 animation: idle, walk, blink, sleep, lie, stretch
  • 성능: 30fps 목표, idle 시 낮은 업데이트 빈도

Behavior Engine

  • finite state machine: idle/walk/sleep/react
  • random weighted scheduler
  • screen bounds collision
  • mouse proximity reactions
  • user activity 기반 졸림/활동성 조정

Settings & Control

  • menu bar/tray icon
  • show/hide companion
  • always-on-top toggle
  • click-through toggle
  • size/speed/personality settings
  • quit / launch at login은 후순위

5. Full Roadmap

Pivot 0

Project Reframe: AI 3D Demo → Screen Friend

목표: README, 앱 copy, docs를 새 제품 목표에 맞게 재정의한다. 기존 upload/generation UI는 future customization prototype으로 내린다.

  • 결과물: 새 README, roadmap, old prototype note.
  • 검증: typecheck/build/diff check.
Slice 1

Character Sandbox in Web

목표: 아직 Electron 없이 웹앱 안에서 캐릭터 행동 loop만 만든다. 화면 아래 stage에서 캐릭터가 idle/walk/blink/sleep/lie 행동을 반복한다.

  • 기존 MacBook 3D viewer는 숨기거나 prototype section으로 이동.
  • 캐릭터 좌표, direction, action state, timer scheduler 구현.
Slice 2

Electron Desktop Shell

목표: apps/desktop 추가. Vite renderer를 Electron 창에 띄운다. 아직 투명/always-on-top 완성 전이라도 desktop app 실행까지 만든다.

  • main process, preload, renderer 연결.
  • dev/build script 정리.
Slice 3

Transparent Always-on-Top Overlay

목표: 실제 Mac 화면 위에 캐릭터만 보이도록 transparent frameless overlay window를 만든다.

  • bottom-of-screen position.
  • setAlwaysOnTop 적용.
  • background transparent.
  • dock/taskbar 노출 최소화 검토.
Slice 4

Click-through & Interaction Mode

목표: 투명 영역은 아래 앱 클릭을 방해하지 않고, 캐릭터 영역만 hover/click 반응 가능하게 만든다.

  • global ignore mouse mode.
  • character interaction toggle.
  • debug overlay로 hitbox 확인.
Slice 5

Behavior Scheduler v1

목표: 캐릭터가 스스로 행동을 고른다. 걷기, 멈추기, 눕기, 졸기, 눈 깜빡임이 자연스럽게 loop된다.

  • weighted random actions.
  • screen edge bounce.
  • idle timeout → sleepy → sleep.
  • mouse near → look/react.
Slice 6

Menu Bar / Tray Settings

목표: 메뉴바에서 show/hide, always-on-top, click-through, size, speed, quit을 조작한다.

Slice 7

Persistence & User Preferences

목표: 사용자가 고른 크기, 위치, 행동 강도, personality를 저장한다.

Slice 8

Character Asset Pipeline

목표: primitive character에서 sprite sheet 또는 GLB asset으로 교체한다. 귀여움/브랜드 감도를 높이는 단계.

Slice 9

Optional AI Custom Character

목표: 사용자가 이미지나 prompt로 캐릭터를 만들 수 있게 한다. 이때 Meshy 같은 image-to-3D API 또는 다른 generation pipeline을 검토한다.

Slice 10

Packaging, QA, Beta

목표: macOS 앱 패키징, signing/notarization 조사, beta feedback, performance QA, crash handling.

8. First Codex Slice Recommendation

바로 Electron을 붙이기 전에, 먼저 현재 repo의 방향을 안전하게 정리하는 Pivot Slice 0를 추천한다.

Recommended model: Codex high / reasoning effort: medium-high

PLAN MODE

We need to pivot the project from an AI 3D MacBook web demo into Screen Friend, a Mac desktop companion app.

Real product goal:
- a cute character lives on the user's actual Mac screen
- always visible on top of normal windows
- appears near the bottom of the screen
- walks, lies down, blinks, gets sleepy, and loops idle behaviors
- later custom AI-generated characters may be supported, but that is not the MVP core

Current repo:
- pnpm workspace
- apps/web Vite + React + TypeScript
- React Three Fiber prototype
- image upload + mock generation flow
- no desktop shell yet

Task:
Inspect the current repo and produce a safe pivot plan only. Do not implement.

Please decide:
1) what to keep
2) what to mark as old prototype/future customization
3) whether to add apps/desktop or convert apps/web
4) first implementation slice after the pivot
5) validation plan

Constraints:
- do not delete existing code
- do not add Electron yet
- do not add dependencies yet
- do not add backend/Meshy/API keys
- keep the plan incremental

10. Market Reference: malang.lab “MacPet” (ADHD 도마뱀)

Instagram 릴 — malang.lab × meshyai_korea 협업 게시물. “맥북에 ADHD 도마뱀 키우기”. 초록 도마뱀 3D 캐릭터가 실제 Mac 화면 위에 살면서 걷고, 벽을 타고, 점프하고, 가끔 춤을 춘다. 게시 2일 만에 좋아요 6,700+ — Screen Friend와 동일 컨셉의 시장 검증 사례.

영상에서 확인된 기술 스택

  • Electron이 아닌 Unity(C#) 기반. 프로젝트명 MacPet.
  • MacOSWindowBridge.cs + MacPetWindow.mm/.bundle: Objective-C 네이티브 플러그인으로 투명/always-on-top 창 제어.
  • URPAlphaClearFeature.cs: URP 렌더 패스에서 알파 클리어로 창 투명화.
  • DesktopPetController.cs: EnterIdle, _dragTarget, SpawnBound 식별자 — idle 상태 머신, 드래그 인터랙션, 화면 경계 처리.
  • 사이드바에 VS CODE PETS 폴더 — 기존 vscode-pets를 레퍼런스로 활용한 정황.
  • Meshy 협업 게시물 — 캐릭터 3D 모델은 Meshy 생성으로 추정.

로드맵 시사점

  • 3D 캐릭터 always-on overlay가 실기기에서 동작 가능함을 실증. 단 Unity 런타임과 Electron+Three.js의 성능 특성은 다르므로 “2D 먼저, GLB는 Slice 8” 전략 유지.
  • Electron transparent window가 한계에 부딪힐 경우 Unity + 네이티브 브리지가 plan B 경로.
  • 벽 타기 · 점프 · 춤은 귀여움 임팩트가 큰 행동 — Slice 5 Behavior Scheduler의 확장 후보로 추가.
  • “ADHD” 같은 personality 프리셋은 Slice 5/7 참고. 기본값은 차분한 행동(리스크 레지스터 참조).
  • Meshy가 desktop pet 방향으로 직접 마케팅 중 — Slice 9(AI 커스텀 캐릭터) 시 협업/API 검토 가치.
